## 3. Feature Encoding

### What was done

**Label Encoding — education_num**
The education column has a natural academic
order from Preschool (1) to Doctorate (16).
This order already exists in education_num so
the raw education column was dropped and
education_num was kept as the ordered encoding.

**One-Hot Encoding — unordered columns**
Seven columns with no natural order were
one-hot encoded using pd.get_dummies():
- workclass
- marital_status
- occupation
- relationship
- race
- sex
- native_country

The first dummy column was dropped using
drop_first=True to avoid multicollinearity.

### Why One-Hot Encoding was used
If label encoding was applied to unordered
columns like workclass, the model would
treat the numbers as having mathematical
meaning. Encoding Private=1, Self-emp=2,
Federal-gov=3 implies Federal-gov is greater
than Private which is false and misleads the
model. One-Hot Encoding creates separate
binary columns with no false ordering.

### Why the first dummy was dropped
When a column has N categories, N binary
columns are created. Only N-1 are needed
because the last category is implied when all
others are zero. Keeping all N columns creates
a perfect linear relationship that confuses
linear models. Dropping the first dummy
avoids this dummy variable trap.

## 4. Train-Test Split and Leak-Free Scaling

### What was done
The data was split into training (80%) and
test (20%) sets. StandardScaler was fitted
only on the training data and used to
transform both sets separately.

### Why train-test split is necessary
The model must be evaluated on data it has
never seen during training. Without a separate
test set it is impossible to know whether the
model has genuinely learned patterns or just
memorised the training data.

### Why only training data was fitted
The scaler computes mean and standard deviation
of each feature. If the scaler is fitted on
the full dataset it uses test set statistics
during training. This means the model
indirectly sees test information before
evaluation giving artificially good results
that would not hold in real production.

Correct approach:
1. Split data first
2. Fit scaler on X_train only
3. Transform X_train and X_test separately

### Output
```
X_train shape            : (26029, 85)
X_test shape             :  (6508, 85)
Train mean after scaling : ~0.000000
Train std after scaling  : ~1.000000
```

### What this means
The training mean is exactly zero and standard
deviation is one confirming the scaler was
applied correctly without data leakage.

## 6. Ridge Regression

### What was done
Ridge Regression was trained with alpha=1.0
using the same train-test split and scaled
features. Results were compared against OLS.

### Why Ridge was applied
OLS Linear Regression can assign very large
coefficients when features are correlated.
Ridge adds an L2 penalty to the loss function
that forces all coefficients to stay small.
This reduces overfitting and produces a more
stable and generalizable model.

### What alpha controls
- alpha = 0   → identical to OLS (no penalty)
- alpha = 1.0 → moderate regularization
- alpha = high → strong shrinkage toward zero

### Ridge vs OLS Comparison
| Model | MSE | R² |
|-------|-----|----|

| Ridge Regression (alpha=1.0) | (your value) | (your value) |

### Why coefficients changed
Ridge shrinks all coefficients toward zero
compared to OLS. Features that had very large
OLS coefficients become smaller under Ridge.
This makes the model less sensitive to any
single feature and more reliable on unseen data.

## 7. Class Imbalance Handling

### Before Balancing
| Class | Count | Percentage |
|-------|-------|------------|
| 0 (<=50K) | 19,769 | 76.00% |
| 1 (>50K) | 6,260 | 24.00% |

### Why imbalance was checked
When one class has significantly fewer samples
the model tends to predict the majority class
most of the time and ignores the minority class.
This gives high accuracy but very poor recall
for the minority class which is unacceptable
for income prediction.

### Why class_weight=balanced was chosen
class_weight=balanced automatically adjusts
the weight of each class inversely proportional
to its frequency. The minority class receives
a higher weight so the model pays more
attention to it during training.

This was preferred over SMOTE because:
- No synthetic data is created
- No risk of overfitting on artificial samples
- Simple to implement
- Equally effective for this dataset size

### Effective Weights Applied
| Class | Weight |
|-------|--------|
| 0 (<=50K) | 0.6579 |
| 1 (>50K) | 2.0789 |

The >50K class gets approximately 3 times
more weight ensuring it contributes equally
to the learning process.

### What each metric means

**Accuracy** is the percentage of all correct

not sufficient here.

were correctly identified. High recall means
fewer missed high earners.

**F1 Score** is the harmonic mean of precision
and recall. It balances both metrics and is
more meaningful than accuracy for imbalanced
datasets.

## 10. ROC Curve and AUC

### Why ROC Curve was generated
The ROC curve shows how well the model separates
the two classes across all possible thresholds.

- X axis: False Positive Rate
  FPR = FP / (FP + TN)

- Y axis: True Positive Rate
  TPR = TP / (TP + FN)

### Precision Formula
```
Precision = TP / (TP + FP)
```

### Recall Formula
```
Recall = TP / (TP + FN)
```

### Which metric is more important
Recall is more important for this task.
Missing a high-income earner (False Negative)
is more costly than a false alarm (False Positive).
The goal is to identify as many >50K earners
as possible.

## 12. Regularization Experiment

### What C controls
C is the inverse of regularization strength.

- High C (C=1.0) → weak regularization
  Model fits data more closely
  Larger coefficients allowed

- Low C (C=0.01) → strong regularization
  Coefficients shrunk toward zero
  Simpler and less flexible model

## 13. Bootstrap Confidence Interval

### Why bootstrap sampling was performed
A single test set evaluation may be influenced
by the specific samples in the test set.
Bootstrap sampling repeatedly draws random
samples and measures performance on each.
This gives a distribution of performance
differences to assess whether C=1.0 consistently
outperforms C=0.01 or whether the difference
is due to chance.

### Method
- Drew 500 bootstrap samples from test set
- Used np.random.choice with replace=True
- Computed AUC(C=1.0) minus AUC(C=0.01)
  for each sample
- Computed mean and 2.5th and 97.5th
  percentiles using np.percentile()

### Results
| Metric | Value |
|--------|-------|
| Mean AUC Difference | 0.0009 |
| 2.5th Percentile | 0.0000 |
| 97.5th Percentile | 0.0018 |
| 95% Confidence Interval | [0.0000, 0.0018] |

### What the confidence interval means
The 95% confidence interval upper bound of
0.0018 is above zero. This means C=1.0
produces a higher AUC than C=0.01 in the
majority of bootstrap samples.

The mean AUC difference of 0.0009 is small
but consistently positive across all 500
samples. This confirms C=1.0 is marginally
but reliably better than C=0.01.

### In simple terms
If these two models were tested on 500
different random samples of similar data
C=1.0 would outperform C=0.01 in the
majority of cases. The advantage is small
but real and not due to random chance.
